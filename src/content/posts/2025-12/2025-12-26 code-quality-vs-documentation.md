쓰레드를 눈팅하다보니, 코드품질과 문서화 관련한 글이 있었다. <br/>
~알고리즘아 제발 개발관련 포스트좀 그만보여줘라~ <br/>
물론 나도 처음엔 저게 무슨소리고 하는 반응이었으나, 하나씩 알아보면서

## 1. 주석사용은 가급적 자제

코드를 읽다보면, 주석은 코드의 옆에 쓰거나 함수 위에 사용하는게 대부분인데 아래처럼

```java
/**
 * - ①
 */
public PostDto getPost(int postId){
    return service.getPost(postId); // - ②
}
```

이러다보면, 코드는 코드대로, 주석은 주석대로 코드를 변경하면 주석도 변경 해야하고 예를들거든 함수가 `deprecated` 혹은 예정이거나 하면 주석에다가도 언제까지다 표기를 해줘야 하고, 아무래도 `코딩 + docs`, `2-track` 형태로 작업형태가 되다보니, 이부분을 신경쓰지 않는다면, 어느한쪽이 구식이 될 가능성이 매우 높다는것

그리고 보통 주석을 다는게 엄청 의미잇는 코드에 다는게 아니다 보니 불필요한 주석이 굉장히 많다는것

## 2. 주석 대신 코드 자체를 이해하기 쉬운 구조로

- 변수 or 함수명을 자연어기반의 의미가 있는편으로
- 자연어 기반으로 함수를 작성하게되면, docs를 읽는것 처럼 코드가 술술 읽히지 않을까 하는

```js
// 주석만 있는 코드
// 이 부분은 파일명에서 확장자를 뽑는 로직입니다
var ext = getExt(file);
var fileExtension = extractFileExtension(file);
```

전자보다, 후자가 영단어 뜻을 알고있다는 전제하에 함수명에서 뜻이 바로 보이니까 주석이 없어도 바로 읽히긴 한다 <br/>
**근데**, 영어권인 곳에서 영어단어를 당연히 의미있게 사용하니까 해석이 되는거라고 생각함

## 3. 주석 대신 코드리뷰, 테스트

### A. 코드 리뷰(Code Review)

- 코드가 왜 이렇게 작성되었는지, 설계 의도, 참고 정보를 리뷰에서 남기고 공유
- 리뷰의견 자체가 문서가 되기도 함 (여러사람이 머리 마주대고 토론한 결과)

### B. 테스트

테스트는 실행 가능한 스펙(명세) 역할을 함

- **테스트의 이름을 문장처럼 쓰기** <br/>
  likeToggle_whenAlreadyLiked_shouldDeleteLikeAndDecreaseCount 서술어를사용하여 함수명을 쓰거나 <br/>
  `@display('설명')` 을 써넣거나
- **Given - When - Then 구조로 의도를 드러내기**<br/>
  Given: 유저 A가 게시글 P를 좋아요하지 않음<br/>
  When: Toggle 호출 <br/>
  Then: like Row 생성, like_count + 1 증가

## 4. 무분별한 주석은 오히려 해가 될 수 있다

- 주석이 **코드보다 빨리 구식이 되어 혼란을 준다**
- 잘못된 주석이 오히려 코드 파악에 방해가 된다
- 주석만 믿고 수정하다가 버그가 생기기도 한다

| 주장                         | 실제상황                                             | 
|---------------------------- | -------------------------------------------------- |
| ✔ 코드에 주석을 전혀 쓰지 않는다   | ✗ 대부분은 필요할 때만 의미 있는 주석을 쓴다                 |
| ✔ 코드 리뷰로 문서화를 대체한다    | ✔ 코드 리뷰에서 의도/설계/컨텍스트 설명을 남기는 경우가 많다     |
| ✔ 코드만 읽으면 모든 정보가 보인다  | ↔ 셀프-도큐먼팅 코드 + 테스트 + 리뷰 + 외부 문서가 함께 쓰임   |
| ✔ 주석 = 나쁜 것               | ✗ 잘 쓰인 주석은 여전히 가치 있음                         |

---

**사례로 이해**

1. 잘못된 주석의 예

```js
while (true) {
  i++; // i 증가
}
```

> 코드 읽으면 **당연한 내용**이라 주석이 소모성 정보가 된다

2. 의미가 있는 주석의 예

```java
// 여기서는 NULL 대신 빈 리스트를 넣는 이유
// => legacy API가 null 처리 로직을 잘못 구현함
```

> 왜 이렇게 코드를 썼는지 설명하고, 향후 **리팩토링 판단 자료**가 된다

## 5. 그렇다면 진짜 문서는 필요없는가?

아니 전혀 그렇지않다고 생각한다

> 시스템의 전체의 흐름, 의도, 목적, 왜이런선택을 해야했는지 코드만으로는 한계가 있음

문서라는게 결국 코딩으로 이뤄진 시스템을 요약한거라 생각하지만, <br/>
먼저 요약본을 보고 코드를 보는게 더 전체적인 틀잡기는 좋을수도 있다는것임

- 함수명 클래스명 자연어기반으로 길게 짓는다고 해결되지 않고
- 주석을 붙이게되면, 코드관리가 어려워지고
- 결국 문서단위로 "서술" 이 가장 효율적인 경우가 많음

## 6. 코드 품질과 문서의 역할은 애초에 다름

여기서 한 번 짚고 가야 할 지점이 있다. <br/>
**코드 품질이 높아진다**는 말과 **문서가 필요 없어질 수 있다는 말**은 같은 축 위에 있지 않다는 점이다.

코드 품질이 좋아진다는 것은 보통 아래를 의미한다.

- 함수와 클래스의 책임이 명확해지고
- 네이밍만 봐도 로직의 역할이 드러나며
- 테스트를 통해 동작이 검증되어 있는 상태

> **이 코드가 무엇을 하는지(what / how)** 는 잘 보이게됨

하지만 문서가 다루는 영역은 다르다.

- 왜 이런 구조를 선택했는지
- 왜 다른 대안이 아닌 이 방식을 택했는지
- 어떤 제약 조건 때문에 이렇게 설계되었는지

> 이건 코드 품질의 문제가 아니라 <br/>
> **시스템 설계와 의사결정의 영역**

아무리 코드가 깔끔해도
왜 이렇게 만들어졌는가 는 코드 안에 자연스럽게 녹이기 어려움

## 7. 그래서 문서는 "대체재"가 아니라 "보완재"

이 글을 쓰면서 정리된 결론은 이것임

> 문서는 코드를 대신하는 게 아니라, <br/>
> 코드가 설명하지 못하는 맥락을 **보완**한다.

- 코드는 **실행 가능한 진실**
- 문서는 **의사결정의 기록**

둘 중 하나가 없어지는 관계가 아니라, <br/> 역할이 다를 뿐이다.

## 8. 개인적으로 정리한 기준

> 6개월 뒤, 이 코드를 처음 보는 사람이 <br/>
> 리뷰 기록 없이 이해할 수 있을까?

Yes - 코드와 테스트로 충분 <br/>
No - 문서로 남기는 게 맞다

## 9. 마무리

`코드 품질이 높아지면 문서는 필요 없어질까?` 라는 질문에 대한 내 답은 명확하다.

**아니다**

다만 문서의 역할은 예전과 달라졌을 뿐임

- 모든 걸 문서로 적는 시대는 지났다
- 하지만 **의사결정과 맥락을 남기는 문서**는 여전히 필요하다
