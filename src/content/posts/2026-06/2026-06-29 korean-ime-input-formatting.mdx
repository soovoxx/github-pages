## 1. 문제가 발생한 상황

기존에 있던 기능이었지만, `Input` 컴포넌트에 휴대폰번호를 입력즉시 포맷팅 하는 기능이 있었는데, 
얼마 전 현업 테스트 중에 꽤 애매한 입력 이슈를 만났다.

예를 들면 이런 흐름이었다.

```tsx
<input
  {...register("phoneNumber")}
  value={value}
  onChange={(e) => {
    const value = e.currentTarget.value;
    const onlyNumber = value.replace(/\D/g, "");
    setValue(formatPhoneNumber(onlyNumber));
  }}
/>
```

처음에는 별문제 없어 보였다. 내 PC에서도 정상적으로 동작했고, 일반적인 테스트 환경에서도 휴대폰 번호 입력은 의도한 대로 처리됐다. 😢 <br/>
그런데 특정 사용자 **노트북**에서만 이상한 현상이 발생했다.

숫자를 입력할 때는 괜찮은데, 숫자가 몇개 입력된 상태에서 한글을 입력시 <br/>
(like. `ㅇ`) 같은 문자를 입력하면 기존에 입력되어 있던 숫자가 하나씩 지워지는 것처럼  보였다.

처음에는 단순한 포맷팅 문제라고 생각했고, `.replace()` 전에 한글이 들어가있는경우, **"포맷팅을 안하면되지"** 않을까 생각을 해서, 그렇게 처리를 해보았으나, 동일했다.

문제는 같은 코드인데도 내 PC에서는 재현되지 않았다는 점이다. **(이게 진짜 미치는 부분)** <br/>
사용자 노트북에서만 발생했고, 테스트 상황에서는 어떻게든 재현을 하고, 문제가 왜 발생되는지를 알아내야 했음,,

찾아보니 IME(Input Method Editor) 조합 이벤트와 관련된 문제라는 것을 알게 되었다.

> - IME란 무엇이며
> - 왜 PC 에서는 발생하지 않고 노트북에서만 발생하는것이었는지
> - 해당문제는 왜 발생하는 것이었는지
> - 수정한다면 어떻게 수정해야 하는지 알아보자

---

## 2. IME는 무엇인가

IME는 Input Method Editor의 약자다.
영문을 제외한 언어를 입력할 때 (일단, 한국어, 중국어, 일본어 다 해당됨), 키보드 입력을 실제 문자로 변환해주는 입력기라고 생각하면 된다.

영어나 숫자는 보통 키 하나를 누르면 바로 문자가 확정된다.
```txt
a
1
```

하지만 한글은 조금 다르다. (with. 일본어, 중국어)

```txt
(한국어 입력 모드)
ㄱ + ㅏ = 가
(일본어 입력 모드)
k → ka → か → カ → 漢字
(중국어 입력 모드)
nihao → 你好
```

이처럼 여러 입력이 조합되어 하나의 글자가 된다. 그래서 한글 입력에는 __"아직 문자가 완전히 확정되지 않은 상태"__ 라는 상태가 별도로 존재한다. <br/>
브라우저는 이 조합 과정을 다루기 위해 `composition event`를 제공한다.

- `compositionStart`
- `compositionUpdate`
- `compositionEnd`

React에서는 주로 `onCompositionStart`, `onCompositionEnd`를 사용해서 조합 중인지 아닌지를 판단할 수 있다.

## 2. compositionEvent와 onChange의 상관관계는 어떻게될까 ?

한글을 입력할 때 이벤트는 대략 이런 흐름으로 발생할 수 있다.

```txt
onChange 
compositionStart
onChange
compositionEnd
onChange
```

여기서 중요한 점은 `onChange`가 조합이 끝난 뒤에만 발생하는 이벤트가 아니라는 것이다.

한글이 아직 조합 중인 상태에서도 `onChange`가 발생할 수 있다.
그리고 이 흐름은 OS, 브라우저, 키보드 드라이버, IME 설정에 따라 조금씩 달라질 수 있다.

각 이벤트의 역할은 다음과 같다.

### compositionStart

문자 조합이 시작될 때 발생한다.

예를 들어 한글 입력 상태에서 `ㄱ`을 입력하기 시작하면, 브라우저는 "지금부터 조합형 문자를 입력하는 중"이라고 판단한다.

### onChange

input의 값이 변경될 때 발생한다.

문제는 한글 조합 중에도 input value가 임시로 바뀔 수 있고, 그때도 `onChange`가 발생할 수 있다는 점이다.

### compositionEnd

문자 조합이 끝나고 최종 문자가 확정될 때 발생한다.

예를 들어 `ㄱ`과 `ㅏ`가 조합되어 `가`가 확정되는 시점이라고 볼 수 있다.

## 3. 어떤 코드에서 문제가 발생했을까

문제가 되었던 코드는 개념적으로 이런 형태였다.

```tsx
const handleChange = (e: React.ChangeEvent<HTMLInputElement>) => {
  const value = e.target.value;
  const onlyNumber = value.replace(/\D/g, "");

  setValue(formatPhoneNumber(onlyNumber));
};
```

휴대폰 번호만 입력받아야 하니, 숫자가 아닌 값은 제거한다.
그리고 다시 휴대폰 번호 형식으로 포맷팅해서 input value에 넣는다.

겉으로 보면 흔한 코드다.

하지만 한글 조합 중에는 이 코드가 문제가 될 수 있다.

예를 들어 input에 이미 이런 값이 들어있다고 해보자.

```txt
010-1234-5678
```

이 상태에서 사용자가 한글 입력 모드로 `ㅇ`을 입력하면, 조합 과정에서 input value가 순간적으로 이런 값처럼 보일 수 있다.

```txt
010-1234-5678ㅇ
```

그런데 `onChange`에서는 바로 숫자만 남긴다.

```tsx
const onlyNumber = value.replace(/\D/g, "");
```

그러면 값은 다시 이렇게 된다.

```txt
01012345678
```

그리고 포맷팅 함수를 거쳐 input value가 다시 세팅된다.

```txt
010-1234-5678
```

여기서 문제가 생긴다.

IME는 아직 `ㅇ`을 조합 중이라고 생각하고 있는데, React controlled input은 value를 다시 렌더링하면서 조합 중인 문자를 제거해버린다.

그 결과 환경에 따라 다음과 같은 현상이 발생할 수 있다.

- 한글 입력이 무시됨
- 커서 위치가 튐
- 마지막 숫자가 지워진 것처럼 보임
- 특정 PC에서는 괜찮은데 특정 노트북에서만 재현됨

내가 겪은 케이스는 마지막에 가까웠다.
일반 PC에서는 괜찮았지만, 특정 노트북에서는 한글 입력 시 기존 숫자가 지워지는 것처럼 보였다.

---

## 4. 어떻게 처리해야 할까

핵심은 단순하다.

> 조합 중에는 포맷팅하지 않고, 조합이 끝난 뒤에 포맷팅한다.

즉, `onChange`에서 무조건 값을 정제하고 포맷팅하는 것이 아니라, 현재 한글 조합 중인지 확인해야 한다.

예시는 다음과 같다.

```tsx
const [isComposing, setIsComposing] = useState(false);

const formatValue = (value: string) => {
  const onlyNumber = value.replace(/\D/g, "");
  return formatPhoneNumber(onlyNumber);
};

const handleChange = (e: React.ChangeEvent<HTMLInputElement>) => {
  const value = e.currentTarget.value;

  if (isComposing || e.nativeEvent.isComposing) {
    setDisplayValue(value);
    return;
  }

  setDisplayValue(formatValue(value));
};
```

그리고 input에서는 composition event를 함께 처리한다.

```tsx
<input
  value={displayValue}
  onCompositionStart={() => {
    setIsComposing(true);
  }}
  onCompositionEnd={(e) => {
    setIsComposing(false);
    setDisplayValue(formatValue(e.currentTarget.value));
  }}
  onChange={handleChange}
/>
```

이렇게 하면 한글 조합 중에는 입력값을 억지로 포맷팅하지 않는다.
그리고 조합이 끝난 시점에 최종 값 기준으로 숫자만 남기고 포맷팅한다.

## 5. 휴대폰 번호 input에서 같이 고려하면 좋은 것들

휴대폰 번호 input이라면 `type="number"`를 사용하는 것도 고민할 수 있다.
하지만 개인적으로는 휴대폰 번호에는 `type="number"`가 잘 맞지 않는다고 생각한다.

휴대폰 번호는 숫자처럼 보이지만 실제로는 숫자 계산을 위한 값이 아니다.

- 앞자리 `0`이 중요하다
- 하이픈이 들어갈 수 있다
- 길이와 포맷이 중요하다
- 숫자 증감 버튼이 필요하지 않다

그래서 보통은 `type="tel"` 또는 `inputMode="numeric"`을 사용하는 쪽이 더 적절하다.

```tsx
<input
  type="tel"
  inputMode="numeric"
  autoComplete="tel"
  value={displayValue}
  onCompositionStart={handleCompositionStart}
  onCompositionEnd={handleCompositionEnd}
  onChange={handleChange}
/>
```

이렇게 하면 모바일에서는 숫자 키패드를 유도할 수 있고, 값 자체는 문자열로 안전하게 다룰 수 있다.

## 6. 정리

이번 이슈는 처음 봤을 때 꽤 당황스러웠다.

같은 코드인데 내 PC에서는 괜찮고, 특정 사용자 노트북에서만 문제가 발생했다.
그리고 숫자만 입력받는 단순한 input에서 한글 입력 때문에 숫자가 지워지는 것처럼 보인다는 점도 직관적이지 않았다.

하지만 원인을 따라가 보니 문제는 입력값 포맷팅 자체라기보다는, 한글 IME 조합 중에 input value를 강제로 바꾸는 흐름이었다.

정리하면 다음과 같다.

- 한글 입력은 조합 과정이 있다
- 조합 중에도 `onChange`가 발생할 수 있다
- `onChange`에서 value를 강제로 정제하고 다시 세팅하면 IME 조합이 깨질 수 있다
- 환경에 따라 특정 PC나 노트북에서만 재현될 수 있다
- 조합 중에는 포맷팅하지 않고, 조합이 끝난 뒤 처리하는 것이 안전하다

처음에는 사용자 노트북의 IME 설정을 바꾸는 방식으로 우회했지만, 이런 종류의 input 컴포넌트를 다시 만들게 된다면 `onChange`만 믿고 처리하지는 않을 것 같다.

입력값을 실시간으로 가공하는 컴포넌트라면, 특히 한글 사용 환경에서는 IME 조합 상태까지 함께 고려해야 한다.
